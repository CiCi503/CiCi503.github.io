浅析volatile

发表于 2017-03-04 | 分类于 Java | | 阅读量

volatile仅仅用来保证该变量对所有线程的可见性，但不保证原子性

Volatile的本意

Volatile的本意是不稳定的，他也就是告诉了编译器，这个变量是不稳定的，是对所有线程共享的，可见的。那么编译器和JVM处理它的时候就会格外小心谨慎。

Volatile做了什么呢

正常变量的基本读取过程

在 java 垃圾回收整理一文中，描述了jvm运行时刻内存的分配。其中有一个内存区域是jvm虚拟机栈，每一个线程运行时都有一个线程栈，线程栈保存了线程运行时候变量值信息。当线程访问某一个对象时候值的时候，首先通过对象的引用找到对应在堆内存的变量的值，然后把堆内存变量的具体值load到线程本地内存缓存中，建立一个变量副本，之后线程就不再和对象在堆内存变量值有任何关系，而是直接修改副本变量的值， 在修改完之后的某一个时刻（线程退出之前），自动把线程变量副本的值回写到对象在堆中变量。这样在堆中的对象的值就产生变化了。

这样带来的问题也显而易见，A线程从堆内存中获取变量对它修改，A线程没有结束那么就不会把变量重新重新放到堆内存中，更新堆内存中变量的值。B线程在A线程没有结束的时候也对同一变量进行获取和修改，那么获取的就是A线程修改前的变量值。两个线程对同一变量同时进行修改就会引起我们所说的并发问题

Volatile变量的读取过程

把变量声明为volatile后啊，JVM就知道了这个变量是不稳定的，是会同时由几个线程访问的。Volatile修饰的成员变量在每次被线程访问时，都强迫从共享内存中重读该成员变量的值。而且，当成员变量发生变化时，强迫线程将变化值回写到共享内存。这样在任何时刻，两个不同的线程总是看到某个成员变量的同一个值。

Volatile有什么用

Java中volatile修饰符是一种用来保证不同线程之间交互的特殊机制。当一个线程修改volatile变量，另一个线程能够看到这个修改。第一个线程通知第二个线程变量已经被修改。

ready是一个volatile boolean变量， 初值设为false，answer是一个非volatile int变量，初值是0.
第一个线程准备好修改ready变量，它是两个线程交流的发送方。第二个线程读取ready变量，获取第一个线程修改的值，因此它是接收方。在两个线程交流时，线程1读取内存中的ready变量，当线程1对它进行修改后，无论该线程是否结束都会将修改后的变量强制输出到内存当中，那么线程2读取的变量就是最新的刚刚修改后的变量值
这就保证了如果线程要输出什么的值的话，那就是42.

如果ready不是volatile的，那会发生什么呢？在线程1和线程2之间将不会有任何明显的交流。
线程1修改ready(现在是non-volatile)后的值可能会传递给线程2，因此线程可能可以读取ready的值true. 然而，线程1也可能不会将修改后的ready的值传递给线程2，answer的值也不会传递给线程2，那么线程2将会输出0.
所以，volatile解决了一个这样的问题：volatile 要求JVM保证线程改变 x 后能立即应用到所有线程，否则有些线程中的x与实际值可能是不同步的（由于缓冲机制）

volatile不能保证原子性

这个很好理解，volatile实现了什么呢，volatile 要求JVM保证线程改变 x 后能立即应用到所有线程，但是同一个x还是可以在多个线程中同时更改，所以可以这样说

  • 对任意单个volatile变量的读/写具有原子性
  • 但类似于volatile++这种复合操作不具有原子性。

总结

当一个变量定义为volatile之后，将具备两种特性：

  1. 保证此变量对所有的线程的可见性，这里的“可见性”，如本文开头所述，当一个线程修改了这个变量的值，新值对于其他线程是立即得知的。但普通变量做不到这点，普通变量的值在线程间传递均需要通过主内存（详见：Java内存模型）来完成。
  2. 禁止指令重排序优化。有volatile修饰的变量，赋值后多执行了一个“load addl $0x0, (%esp)”操作，这个操作相当于一个内存屏障（指令重排序时不能把后面的指令重排序到内存屏障之前的位置），只有一个CPU访问内存时，并不需要内存屏障；（什么是指令重排序：是指CPU采用了允许将多条指令不按程序规定的顺序分开发送给各相应电路单元处理）。

综上，volatile仅仅用来保证该变量对所有线程的可见性，但不保证原子性。

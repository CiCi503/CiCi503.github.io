Spring IOC

Spring IOC是干啥的

IOC，Inversion Of Control，字面上理解是控制反转，是什么的控制被反转了呢，是获得对象对象的过程被反转了。原来我们是怎样获取对象的呢，我们需要一个对象的时候就new一个对象出来，是自己生产对象。那IOC是怎样获取对象呢，是有一个对象工厂BeanFactory，负责生产对象，当我们需要对象的时候，从BeanFactory中拿过来用。

DI，Dependency Injection，依赖注入，指的是一个类A中有成员变量是B类的对象，那么怎么得到这个对象呢，原来我们是怎么做的啊，是需要这个对象就new出来嘛。但是DI是由IOC容器在运行期间，动态地将某种依赖关系注入到对象之中。Spring有两种注入方式，设值注入和构造注入，稍后我们会详细介绍。

Spring IOC怎么用

知道了IOC是啥以后，我们就要关心怎么使用的问题了。有两种方式，一种是通过XML文件的配置，另一种是通过注解来配置。

Spring容器

Spring容器是什么

Spring容器是生成Bean的工厂，并管理容器中的Bean。Bean是Spring管理的基本单位

Spring有两个核心接口，BeanFactory和ApplicationContext，其中ApplicationContext是BeanFactory的子接口，也被称为Spring上下文。他们都可以代表Spring容器

BeanFactory的基本方法
传入Xml文件

创建Spring容器的实例时，要提供Spring容器管理的详细配置信息。Spring的配置信息采用XML配置文件来设置。XML文件可以用以下几种方式传入。

  1. 实例化BeanFactory

  1. 实例化ApplicationContext

    1. ApplicationContext通常有两个实现类：

      1. FileSystemXmlApplicationContext(以基于文件系统的XML配置文件创建ApplicationContext实例)
      2. ClassPathXmlApplicationContext：以类加载路径下的XML配置文件创建ApplicationContext实例

    2. ApplicationContext可以同时加载多个配置文件

ApplicationContext相对于 BeanFactory的额外特性
  • ApplictionContext作为Spring容器，创建容器时会同时创建容器中所有的singleton作用域Bean，因此会需要更多的系统开销。但一旦创建成功，后面应用的响应速度更快。
  • Application提供国际化支持：ApplicationContext接口继承了MessageSource接口，因此具备国际化功能
  • Application的事件机制：有一个容器事件继承ApplicationEvent类，一个坚挺其继承ApplicationListener，每当ApplicationContext发布ApplicationEvent时，ApplicationListener都会被触发

一个类里实现了BeanFactoryAware接口，那么当这个类的Bean实例被容器创建以后，它会拥有一个引用，指向创建它的BeanFactory

Spring的常用注入方式

Spring的注入

Spring的注入是指在启动Spring容器加载bean配置的时候，当某个Java实例需要另一个Java实例时，由Spring容器动态地注入到第一个Java实例中

常用的注入方式有两种，设值注入，构造注入

设值注入
  • 设值注入是在需要其他Java实例的类中通过Setter()方法来注入被依赖的实例。

举个栗子：我们有两个类，AutoWiringDao，AutoWiringService，AutoWiringService中有一个AutoWiringDao类的对象

我们发现，我们从来没有手动地将AutoWiringService中的AutoWiringDao的对象实例化，却成功调用了该对象的方法，这就是依赖注入DI(Dependency Injection)，也就是在程序运行的过程中，如果需要另一个对象协作，那么无需再代码中创建被调用者，而是在外部注入这种依赖

IoC的一个重点是在系统运行中，动态的向某个对象提供它所需要的其他对象。这一点是通过DI（Dependency Injection，依赖注入）来实现的。

构造注入
  • 设值注入是在需要其他Java实例的类中通过构造方法来注入被依赖的实例。
两种方法的比较
  • 设置注入与传统的JavaBean写法很相似，更容易理解，接受。
  • 对于复杂的依赖关系，如果采用构造注入，构造器会很臃肿
  • 所以，建议使用设值注入

Spring Bean

元素是Spring配置文件的根元素，可以指定如下属性

  • default-lazy-init： 指定该元素下配置的所有Bean默认的延迟初始化行为
  • default-merge
  • default-autowire
  • default-autowire-candidate
  • default-init-method
  • default-destroy-method
    元素上面的属性都可以在子元素中指定，将属性名去掉default即可。在中制定的属性只对特定的Bean起作用，在中定义的属性对所有的Bean都起作用
    Bean的配置项
  • Id
  • Class
  • Scope
  • Constructor arguments
  • Properties
    • value 用于指定字符串类型/基本类型的属性值
    • ref 要为Bean注入的属性值是容器中另一个Bean实例
    • list，set，map，props 指定注入属性是集合
  • Autowiring mode
  • lazy-initialization mode
  • Initialization/destruction method
Bean的作用域
  • singleton: 单例，指一个Bean容器中只存在一份
  • prototype: 每次请求（每次使用）创建新的实例
  • request: 每次http请求创建一个实例且仅在当前request内有效
  • session: 同上，每次http请求创建，当前ssession有效
  • global session: 基于prtlet的web中有效（portlet定义了global session），如果是在web中，同session
Bean的生命周期
  • 定义：在XML文件中定义了id,class
  • 初始化: IOC容器启动时加载配置文件的bean，生成具体的实例
    • 实现org.springframework.beans.factory.InitializaingBean接口，覆盖afterPropertiesSet方法
    • xml配置init-method
  • 使用：在单元测试中，在实际开发中取出bean容器建立对象
  • 销毁：在bean容器停止的时候销毁由当前bean容器创建的所有的实例
    • 实现org.springframework.beans.factory.DisposableBean接口，覆盖destroy-method方法
    • xml配置destroy-method

也可以配置全局默认的初始化，销毁方法，在xml文件的最外层配置

关于bean初始化和销毁同时使用的注意情况：

  1. 默认全局的初始化和销毁方法；
    default-init-method="init" default-destroy-method="destroy"

  2. 实现接口的初始化和销毁方法；
    public class BeanLifeCycle implements InitializingBean, DisposableBean

  3. 配置文件中配置初始化和销毁方法；

这三个方法同时使用时，1默认的则不执行，而23两种都会执行，并且是2实现接口的方式先于配置中3的执行。

1默认的全局初始化和销毁方法可以有可以没有，有没有对配置都没有太大影响，当然如果一个bean没有采取23初始化销毁方法，而有1默认的方法的话，这两个方法还是会执行的。即使没有，系统也不会报错。

通过注解配置

ClassPath扫描与组件管理

从Spring3.0开始，Spring JavaConfig项目提供了很多特性，包括使用Java而不是XML定义bean，也就是说不用XML文件，而是用Java语言本身来定义bean。那是怎么弄呢？就是基于注解的方式。比如@Configuration，@Bean，@Import，@DependsOn

  • @Component是一个通用注解，可用于任何bean
  • @Repository，@Service，@Controller是更有针对性的注解
    • @Repository通常用于注解DAO类，即持久层
    • @Service通常用于注解Service类，即服务层
    • @Controller通常用于Controller类，即控制层（MVC）
元注解（Meta-annotations）
  • 许多Spring提供的注解可以作为自己的代码，即“元数据注解”，元注解是一个简单的注解，可以应用到另一个注解
类的自动检测及Bean的注册
  • Spring可以通过检测注解自动检测类并注册Bean到ApplicationContext中，只需要将注解放到类上面，或者方法上面
  • 为了能够检测这些类并注册相应的Bean，需要在xml中配置<context:conponent-scan base-package = "在哪个包中"/>。context：component-scan包含context：annotation-config，通常使用前者，不再使用后者
使用过滤器进行自定义扫描
  • 默认情况下，类被自动发现并注册bean的条件是：使用@Component，@Repository，@Service，@Controller注解或者使用@Component的自定义注解
  • 可以通过过滤器修改上面的行为，例如，下面例子的XML配置忽略了所有的@Repository注解并用“Stub”代替
  • 还可以使用use-default-filters = "false" 禁用自动发现与注册
说了辣么多，怎么用呢

Easy啦，在要生成bean的类前加上注解，然后在xml配置文件中加上<context:conponent-scan base-package = "在哪个包中"/>。

@Required注解
  • @Required注解使用与bean属性的setter()方法
  • 这个注解仅仅表示，受影响的bean属性必须在配置时被填充，通过在bean定义或通过自动装配一个明确的属性值
@Autowired注解
  • 可以将@Autowired注解为传统的setter方法
  • 可用于构造器或成员变量（用在哪就放哪的脑袋顶上就行啦）
  • 默认情况下，如果找不到合适的bean将会导致autowiring失败抛出异常，可以通过设置@Autowired(required = false)来避免，也就是说这不是一个必需的，找不到也没关系
  • 每个类只能有一个构造器被标记为required=true
  • @Autowired的必要事项，建议使用@Required注解

  • 可以使用@Autowired注解那些众所周知的解析依赖性接口，比如：BeanFactory，ApplicationContext，Environment，ResourceLoader，ApplicationEventPublicsher，and MessageSource

  • 可以通过添加注解给需要该类型的容器，以提供ApplicationContext中的所有特定类型的bean。说人话就是在一个Map/Set头顶上加@Autowired，在实体类脑袋顶上加@Component，就把@Component类生成的那些bean都加到容器中了

  • 如果希望使用的list/数组有序，可以让bean实现org.springframework.core.Ordered接口或使用@Order注解，例如@Order(1)
Quallifier注解
  • 按类型自动装配可能多个bean实例的情况，可以使用Spring的@Qualifier注解缩小范围，也可以用于指定单独的构造器参数或方法参数
  • 可用于注解集合类型变量
  • @Autowired适用于fields，constructorscrfjdi
基于Java的容器注解
  • @Bean标识一个用于配置和初始化一个由SpringIoC容器管理的新对象的方法，类似于XML配置文件的<bean/
  • 可以在Spring的@Component注解的类中使用@Bean注解任何方法
  • 在上一点中通常使用的是@Configuration
